# 产品路线图 (ROADMAP)

本文档描述项目从 V1 到 V4 的演进路线, 以及每个版本的核心交付物。

## V1.0 (当前版本) — 城市级公司作息地图

### 核心交付

- Excel 上传 (`.xlsx` / `.xls`), 浏览器本地解析
- 自动识别 955 / 965 / 996 三个数据区域
- 自动多城市拆分 (`"北京/上海"` → 两条记录)
- 自动分类 `work_system` / `weekend_type` / `risk_level`
- 城市级聚合 + 强度评分 (0-100)
- SVG 交互式中国地图 (省界 + 城市点位 + 缩放/平移)
- 点击城市 → 公司列表 → 公司详情
- 多维度筛选 (工作制度 / 周末类型 / 工作强度等级 / 可信度 / 城市)
- 关键词搜索 (公司名 / 城市 / 规则 / 证据)
- 多格式导出 (`normalized_companies.json` / `city_summary.json` / `map.geojson` / `.csv`)
- 完整免责声明与"非绝对化"措辞

### 已知限制

- 仅城市级坐标, 无法看到公司具体办公点
- 证据字段多为描述性文本, 缺少可访问 URL
- 无用户投稿与审核机制
- 无招聘岗位 / 薪资信息
- 无时间序列 (无法看到公司作息随时间的变化)

### MVP 成功标准

- [x] 上传示例 Excel 后, 5 秒内地图渲染完成
- [x] 城市点位颜色与工作强度等级一致
- [x] 点击城市能看到该公司列表
- [x] 点击公司能看到规则和证据原文
- [x] 筛选 996 后, 地图上 955 城市点消失
- [x] 搜索"拼多多"能定位到对应记录
- [x] 导出 JSON / GeoJSON / CSV 文件可被其他工具加载

## V1.5 — 地址补全 (计划中)

### 核心交付

- 用户手动补充公司地址 (输入框 + 高德/百度地理编码 API)
- 城市内公司点位地图 (MapLibre GL JS + deck.gl 替换 SVG)
- 公司点位聚合 ( Hexbin / Cluster )
- 沿地铁线查看公司作息 (地铁数据接入)
- 公司周边通勤分析 (1km / 3km / 5km 半径内的公司密度)

### 关键决策

- **是否引入 MapLibre?** 是。V1.5 数据量从 38 城市点位扩展到 300+ 公司点位, SVG 性能不足。
- **地理编码用谁?** 优先高德 Web 服务 API (免费配额高), 备选百度地图 API。
- **地址粒度?** 优先精确到写字楼, 退而求其次精确到街道。

## V2.0 — 用户投稿与审核 (计划中)

### 核心交付

- 用户投稿表单 (公司 / 城市 / 工作制度 / 规则 / 证据截图上传)
- 投稿审核后台 (管理员审批 / 拒绝 / 标记存疑)
- 证据图片上传 (OSS / S3 存储, 自动压缩 + 水印)
- 可信度评分细化 (基于: 投稿人信誉 + 证据数量 + 时间新鲜度 + 多源验证)
- 评论系统 (用户对每条记录补充信息或质疑)
- 时间序列 (公司作息随时间的变化曲线)

### 关键决策

- **是否需要登录?** 投稿需要 (邮箱验证 + 可选手机验证), 浏览不需要。
- **如何防滥用?** 投稿频率限制 + IP 限流 + 反垃圾关键词。
- **如何处理争议?** 公司方可申请反驳, 反驳内容附在记录下方。

## V3.0 — 招聘岗位地图 (计划中)

### 核心交付

- 公开招聘网页导入 (Boss 直聘 / 拉勾 / 智联 / 51job 等公开页)
- 岗位字段: 岗位名 / 薪资范围 / 经验要求 / 学历要求 / 福利标签
- 单双休关键词提取 (从职位描述中识别"双休"/"大小周"/"单休"/"996"等)
- 公司作息 ↔ 招聘信息关联 (同一公司多维度展示)
- 岗位热度地图 (按城市 / 岗位类型展示需求量)
- 薪资-作息散点图 (高薪是否=996?)

### 关键决策

- **数据来源合规?** 仅使用公开网页, 不绕过验证码, 不采集个人信息 (简历 / 手机号 / 聊天记录)。
- **更新频率?** 周更 (避免对招聘网站造成压力)。
- **岗位过期?** 30 天后自动标记为"可能已关闭"。

## 不在路线图中的功能 (明确拒绝)

以下功能**不会**实现, 因为它们与项目定位不符或存在法律/道德风险:

| 功能              | 拒绝原因                                           |
| ----------------- | -------------------------------------------------- |
| 简历投递          | 本项目是数据可视化工具, 不是招聘平台               |
| 公司评分          | 主观评分容易被刷, 且偏离"作息数据"的核心           |
| 个人吐槽/爆料     | 法律风险高, 且无法验证                              |
| HR 联系方式       | 涉及个人信息, 不采集                               |
| 加班打卡记录上传  | 涉及个人考勤数据, 不采集                           |
| 公司市值/财报数据 | 与作息无直接关系, 增加复杂度                        |
| 匿名聊天室        | 容易演变成情绪宣泄场所, 偏离数据产品定位            |

## 优先级判断原则

每个新版本的功能优先级, 按以下原则排序:

1. **数据真实性** > 数据完整性 > 数据新鲜度 > 数据广度
2. **核心用户场景** (查看公司作息) > 辅助场景 (投稿 / 评论)
3. **离线可用** > 在线可用 (V1 全程无外部依赖)
4. **隐私保护** > 功能丰富度
5. **代码可维护** > 短期效率

## 如何贡献

欢迎通过 Issue / PR 贡献:

- 报告数据错误 (城市坐标错误、分类错误)
- 补充城市坐标库 (`src/lib/city-centers.ts`)
- 改进分类规则 (`src/lib/classify.ts`)
- 添加新的导出格式
- 完善文档

**不接受**以下 PR:
- 引入爬虫功能的 PR
- 引入用户登录 + 评论系统的 PR (在 V2.0 才会做)
- 修改免责声明措辞使其更"绝对化"的 PR
